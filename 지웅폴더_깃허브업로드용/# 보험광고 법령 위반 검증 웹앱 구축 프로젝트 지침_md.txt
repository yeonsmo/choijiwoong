# 보험광고 법령 위반 검증 웹앱 구축 프로젝트 지침

## 0. 프로젝트 개요 및 절대 원칙

너는 보험광고 콘텐츠의 법령 위반 여부를 검증하고, 법령 준수 콘텐츠를 생성하는 웹 애플리케이션을 구축한다. 이 애플리케이션은 Vercel에 배포되며, 사용자가 브라우저 URL로 접속하여 사용하는 웹앱이다. 데스크톱 설치형이 아니다.

### 0-1. 절대 준수 원칙
1. 이모지를 어떠한 출력에도 사용하지 않는다.
2. 모든 비밀 키(법제처 API 키, AI API 키, DB 자격증명)는 클라이언트에 노출하지 않는다. 서버 사이드(Vercel Serverless Functions, 환경변수, 암호화된 DB 컬럼)에만 저장한다.
3. 미구현 또는 동결 상태의 기능은 임의로 삭제하거나 생략하지 않는다. 명시된 동결 방식대로 코드를 작성하되 비활성 상태로 둔다.
4. 사용자가 결정하지 않은 사항을 임의로 추측하여 하드코딩하지 않는다. 불명확한 지점은 환경변수 또는 설정 메뉴로 외부화한다.

## 1. 기술 스택 및 배포 환경

1. 프레임워크: Next.js (App Router 기반)를 사용한다.
2. 배포: Vercel에 배포한다.
3. 데이터베이스: Supabase 무료 티어로 시작한다. PostgreSQL 기반이며, 추후 사용자 본인 DB 계정으로 전환 가능하도록 DB 연결 정보를 환경변수로 완전히 외부화한다.
4. 인증: Supabase Auth를 사용한다.
5. 파일 저장: 영상 등 대용량 파일은 Vercel Serverless의 페이로드 제한(4.5MB)과 실행 시간 제한을 우회하기 위하여 Cloud Storage(Vercel Blob 또는 Supabase Storage)에 직접 업로드한 후 그 URL을 분석 엔진에 전달하는 구조로 설계한다.

## 2. 인증 및 권한 관리 체계

### 2-1. 권한 등급 정의
1. 최고권한자(Super Admin): 모든 기능을 제어, 통솔, 설정할 수 있는 단일 최상위 권한자다. 프로젝트 소유자 본인이다.
2. 관리자(Admin): 신규 계정 생성, API 키 입력 등 운영에 필요한 모든 권한을 보유한다. 단, 최고권한자에 대한 제어 권한은 없다.
3. 일반 사용자(User): 분석, 생성, 다운로드 등 일반 기능만 사용 가능한 기본 등급이다.

### 2-2. 로그인 페이지
1. 이메일/비밀번호 기반 로그인 페이지를 구현한다.
2. 인증되지 않은 사용자는 모든 기능 페이지에 접근할 수 없으며, 로그인 페이지로 리다이렉트된다.

### 2-3. 계정 생성 방식 (폐쇄형)
1. 일반 공개 회원가입 기능을 구현하지 않는다.
2. 신규 계정은 관리자(Admin) 또는 최고권한자(Super Admin)만 생성할 수 있다.
3. 계정 생성 시 권한 등급(User, Admin)을 지정할 수 있다. 단, Super Admin 등급은 신규 생성으로 부여할 수 없으며 초기 시드 계정으로만 존재한다.

### 2-4. 관리자 페이지
1. Admin 이상 권한자만 접근 가능한 별도 관리자 페이지를 구현한다.
2. 관리자 페이지에서 수행 가능한 작업: 계정 목록 조회, 신규 계정 생성, 계정 권한 등급 변경, 계정 비활성화 및 삭제, 시스템 설정 관리.

### 2-5. 최고권한자 보호 규칙 (서버 사이드 강제)
관리자(Admin)가 시도하더라도 다음 4가지 행위는 서버 사이드 검증 로직에서 반드시 차단한다. 클라이언트 측 UI 숨김만으로 처리하지 않고, API 레벨에서 권한을 검증하여 거부한다.
1. 최고권한자 계정 삭제 차단.
2. 최고권한자 권한 등급 변경 차단.
3. 최고권한자 비밀번호 강제 변경 차단.
4. 최고권한자가 입력한 시스템 설정(법제처 API 키 등 핵심 설정 포함)의 임의 변경 차단.

## 3. 법령 데이터 수집 모듈

### 3-1. 법제처 API 연동
1. 국가법령정보 공동활용(법제처 OPEN API)을 연동하여 법령 데이터를 수집한다.
2. 법제처 API 키는 설정 메뉴에서 입력하며, 교체, 삭제 후 재등록이 가능하다.
3. 법제처 API 키 관리 기능은 아래 3-3의 91일 업데이트 카운터와 완전히 독립적으로 동작한다. 키를 교체하거나 삭제하더라도 카운터에 영향을 주지 않으며, 카운터 동작이 키 관리에 영향을 주지 않는다.

### 3-2. 수집 대상 법령 범위
다음 7개 범주의 법령 및 규정을 다중적이고 방대하게 수집하여 DB에 저장한다.
1. 보험업법 및 동법 시행령, 시행규칙.
2. 표시·광고의 공정화에 관한 법률.
3. 금융소비자 보호에 관한 법률.
4. 금융위원회 보험업감독규정.
5. 보험업감독업무시행세칙.
6. 관련 행정규칙 및 고시.
7. 공정거래위원회 표시·광고 심사지침.

### 3-3. 91일 업데이트 카운터 및 팝업
1. 최초 데이터 수집일을 DB에 저장한다.
2. 프로그램 실행(접속) 시마다 현재 날짜와 최초 수집일(또는 최종 수집일)을 비교한다.
3. 91일이 경과한 시점에 자동으로 팝업창을 띄운다. 팝업 문구 예시: "데이터를 업데이트 하시겠습니까?"
4. 사용자가 "예"를 클릭하면 법령 데이터를 재수집하고 수집일을 갱신한다. "아니요"를 클릭하면 수집하지 않고 닫는다.

### 3-4. 사전 세팅 및 키 회수 시나리오
1. 프로젝트 소유자는 전달 전 본인의 법제처 API 키를 입력하여 법령 데이터를 사전에 다방면, 다중적, 방대하게 수집해둔다.
2. 사전 수집 완료 후 소유자 본인의 법제처 API 키는 삭제한다.
3. 전달받는 사용자가 추후 본인의 법제처 API 키를 설정 메뉴에서 입력하여 업데이트 기능을 사용한다.
4. 따라서 키가 삭제된 상태에서도 기존 수집 데이터로 분석 기능이 정상 동작해야 하며, 업데이트만 키 재입력 후 가능하도록 설계한다.

## 4. AI API 키 관리 및 분석 엔진

### 4-1. AI API 키 관리 (설정 메뉴)
1. AI API 키 입력란은 설정 메뉴에 별도로 둔다.
2. 다수의 AI API 키를 입력할 수 있도록 추가 기능과 삭제 기능을 제공한다.
3. 최초 기본 입력란으로 Gemini(Vision LLM) 키 입력란을 제공한다.
4. 모든 AI API 키는 서버 사이드에 암호화하여 저장한다.

### 4-2. 분석 엔진 동작 (단일 모델)
1. 사용자가 사진 또는 영상 파일을 업로드한다.
2. 영상은 Cloud Storage에 업로드한 후 URL을 전달하는 방식으로 처리한다(2-1, 1-5 참조).
3. Vision LLM이 콘텐츠를 분석하고, 3장에서 수집한 법령 DB와 대조하여 위반 또는 위반하지 않음을 판별한다.
4. AI API 키가 1개만 입력된 경우 해당 단일 모델만 구동한다.

### 4-3. 토론 오케스트레이션 (다중 모델, Cross-Examination 방식)
1. AI API 키가 2개 이상 입력되면 자동으로 토론 오케스트레이션 구조로 확장한다.
2. 토론 방식은 Cross-Examination(교차 검증) 방식으로 구현한다.
   - 1단계: 한 모델이 1차 판별을 수행한다(위반 여부 및 근거 제시).
   - 2단계: 다른 모델이 1차 판별 결과를 비평하고 반론 또는 보강한다.
   - 3단계: 비평을 반영하여 재판별하고 최종 결론을 도출한다.
3. 입력된 키 개수에 따라 참여 모델이 동적으로 확장된다. 키가 추가될수록 교차 검증에 참여하는 모델이 늘어나는 구조로 설계한다.
4. 최종 판별 결과에는 각 모델의 의견, 비평 내용, 최종 합의 결론, 근거 법령 조항을 함께 표시한다.

## 5. 생성 모델 (콘텐츠 생성 기능)

### 5-1. 생성 입력 및 기본 동작
1. 사진, 영상, 데이터를 입력하면 AI 에이전트가 법령 DB를 참고하여 법령 준수를 기초로 콘텐츠를 생성한다.
2. 생성 시 마케팅 트렌드와 시각적 효과를 사람들의 선호도에 맞춰 반영하도록 설계한다.

### 5-2. 출력 유형 선택 및 API 키 매핑
사용자가 출력 유형을 선택할 수 있도록 하며, 각 유형에 맞는 API 키가 입력되어 있을 때만 해당 기능이 작동한다.
1. 이미지 생성: OpenAI DALL-E, Google Imagen, Stability AI 중 선택. 해당 키 입력 시 작동.
2. 카피라이팅 생성: Gemini, OpenAI GPT, Anthropic Claude 등 LLM API. 해당 키 입력 시 작동.
3. 영상 생성: Runway, Pika, Google Veo 등. 별도 키 입력란을 제공하며 해당 키 입력 시 작동.
4. 특정 유형의 API 키가 입력되지 않은 경우 해당 출력 유형은 비활성 상태로 표시하고, 키 입력을 안내한다.

### 5-3. 마케팅 트렌드 데이터 연동
1. 기본 동작: LLM의 내재 지식을 활용하여 트렌드를 반영한다.
2. 확장 기능: 외부 트렌드 API를 연동할 수 있는 범용 구조를 구현한다. 구성 요소는 다음과 같다.
   - 트렌드 API 키 입력란.
   - 엔드포인트 URL 입력란.
   - 응답 파싱 규칙 설정란(어떤 API든 연결 가능하도록 범용 설계).
3. 외부 트렌드 API가 설정되면 내재 지식 대신 또는 함께 외부 데이터를 반영한다.

## 6. 순환 워크플로우 (생성 후 재검증)

1. 5장에서 생성된 콘텐츠는 다운로드가 가능하다.
2. 생성된 콘텐츠의 위반 여부 판별은 사용자의 책임과 몫임을 명시한다.
3. 사용자가 메뉴를 클릭하면 생성된 콘텐츠를 4장의 분석 엔진(단일 모델 판별 또는 다중 모델 토론 오케스트레이션)으로 다시 검증하는 순환 워크플로우로 이어진다.
4. 즉, 생성 → 다운로드 → 재검증 → 결과 표시가 하나의 순환 구조로 연결된다.

## 7. 웹훅 및 외부 API 발행 모듈

1. 확장 및 고도화를 위하여 이 애플리케이션 자체가 웹훅 또는 API 키를 발행할 수 있는 기능을 구현한다.
2. 발행된 키로 외부 시스템이 이 애플리케이션의 분석 기능 등을 호출할 수 있도록 인증 메커니즘을 설계한다.
3. 키 발급, 조회, 폐기 기능을 관리자 페이지에 둔다.

## 8. 인스타그램 자동 업로드 모듈 (백엔드 완전 구현, 동결 상태)

### 8-1. 백엔드 구현 범위 (동작은 안 하지만 실제 호출 가능한 완전 함수)
1. Instagram(Meta) Graph API 연동을 위한 완전한 함수를 구현한다. 포함 요소: 인증 토큰 관리 함수, 미디어 업로드 컨테이너 생성 함수, 게시(publish) 함수, 스케줄러 로직(설정된 시간 또는 AI 에이전트가 사전 조사하여 결정한 적절한 시간에 자동 업로드).
2. 이 함수들은 실제 호출 가능한 완전한 형태로 작성하되, 실제 Meta 액세스 토큰 등 자격증명은 환경변수에 입력되지 않은 동결 상태로 둔다.
3. 환경변수에 자격증명을 추가하고 메뉴 토글을 활성화하는 것만으로 즉시 동작하도록 설계한다.

### 8-2. 프론트엔드 표시
1. 프론트엔드에 인스타 자동 업로드 기능을 별도 메뉴로 표시한다.
2. 해당 메뉴에 진입하면 "현재 구현되지 않은 기능입니다"라고 표시한다.
3. 프로젝트 소유자가 원할 때 언제든 활성화할 수 있도록, 백엔드 함수와 프론트엔드 메뉴를 연결만 하면 동작하는 구조로 설계한다.

### 8-3. Meta API 외부 제약 명시
1. Instagram Graph API는 비즈니스 또는 크리에이터 계정, Facebook 페이지 연동, Meta 앱 심사 통과가 필요하다는 사실을 코드 주석 및 안내 문구에 명시한다.

## 9. 출력 및 보고 규칙

1. 모든 출력은 사실과 수치 중심으로 작성한다.
2. 이모지를 사용하지 않는다.
3. 위반 판별 결과는 위반 여부, 근거 법령 조항, 판별 신뢰도, 참여 모델 의견을 명확히 구분하여 표시한다.
4. 오류 발생 시 임의 복구를 시도하지 않고 오류 원문을 표시한다.

## 10. 구현 우선순위 (Step-by-Step)

다음 순서로 구현한다.
1. Next.js + vercel + DB 기반 인프라 및 환경변수 구조 구축.
2. DB 기반 로그인 및 3등급 권한 체계, 관리자 페이지, 최고권한자 보호 규칙 구현.
3. 법제처 API 연동, 법령 데이터 수집, DB 저장, 91일 카운터 및 팝업, 키 관리(카운터 독립).
4. AI API 키 관리(Gemini 기본, 다중 추가/삭제) 및 단일 모델 분석 엔진.
5. 다중 모델 토론 오케스트레이션(Cross-Examination) 구현.
6. 생성 모델(출력 유형 선택, API 키 매핑, 트렌드 API 범용 연동) 구현.
7. 순환 워크플로우(생성 후 재검증) 연결.
8. 웹훅 및 외부 API 발행 모듈 구현.
9. 인스타 자동 업로드 백엔드 완전 함수 구현 및 프론트엔드 동결 표시.
10. 통합 테스트 및 무료 티어 정상 작동 확인.

# 아이콘 셋업 작업 지시

## 전달 파일
- 원본 아이콘: [파일명].svg (또는 고화질 PNG)

## 작업 목표
이 Next.js(App Router) 프로젝트에 두 가지 용도의 아이콘을 정확한 위치와 형식으로 배치한다.
(1) 브라우저 탭 파비콘, (2) PWA 홈 화면 설치용 앱 아이콘.

## 1. 파비콘 (브라우저 탭 상단)
1. 전달받은 SVG를 app/icon.svg 위치에 배치한다. Next.js App Router가 자동 인식한다.
2. 구형 브라우저 호환을 위해 app/favicon.ico (32x32)도 함께 생성한다.
3. 별도의 <link> 태그 수동 삽입은 하지 않는다. App Router의 파일 컨벤션에 의존한다.

## 2. PWA 앱 아이콘 (홈 화면 설치 시)
1. 원본 SVG로부터 다음 PNG 크기를 파생 생성하여 public/icons/ 폴더에 저장한다:
   - icon-192.png (192x192)
   - icon-512.png (512x512)
   - icon-180.png (180x180, iOS apple-touch-icon용)
   - icon-512-maskable.png (512x512, maskable purpose용, 여백 안전영역 적용)
2. public/manifest.json (또는 app/manifest.ts)을 생성하고 위 아이콘을 등록한다.
   - name, short_name, start_url("/"), display("standalone"), background_color, theme_color, icons 배열 포함.
   - icons 배열에 192, 512(purpose: "any"), 512-maskable(purpose: "maskable")를 명시한다.
3. iOS 대응: app/layout.tsx의 metadata에 apple-touch-icon(icon-180.png)을 등록한다.

## 3. 검증
1. 빌드 후 브라우저 탭에 파비콘이 정상 표시되는지 확인한다.
2. 크롬 개발자도구 > Application > Manifest 탭에서 manifest.json이 오류 없이 로드되고 아이콘이 모두 인식되는지 확인한다.
3. 모바일에서 "홈 화면에 추가" 시 설치 아이콘이 정상 표시되는지 확인 절차를 보고한다.

## 금지 사항
1. 이모지를 사용하지 않는다.
2. 아이콘 파일을 임의로 디자인하거나 변형하지 않는다. 전달받은 원본을 크기 변환만 한다.
3. 확정되지 않은 name, theme_color 등은 임의 입력하지 않고 사용자에게 값을 질의한다.
